# Web Clip 記事整理スキル

Obsidian Web Clipperで保存されたMarkdown記事を読み取り、
内容を解析してカテゴリ別に自動整理する。

## パス設定

各パスはプロジェクトのルートからの相対パスで指定する。

- **入力元**: `./Clippings`
- **整理先**: `./04_Articles`
- **カテゴリ分類先**: `./03_MOC`
- **画像保存先**: `./01_Assets`
- **画像取得スクリプト**: `./.claude/skills/clip/scripts/fetch_images.py`
- **画像メタ情報保存パス**: `./.claude/skills/clip/scripts/tmp/images.json`
- **画像情報保存パス**: `./.claude/skills/clip/scripts/tmp/`

## ワークフロー

### 1. 未整理ファイルの検出

入力元フォルダ内の `.md` ファイルを全て取得する。
ファイルが0件なら「整理対象の記事はありません」と報告して終了。

手順2以降を取得したファイル毎に繰り返す。

### 2. ファイル内容の読み取り

- mdファイルを読み取り、内容を解析する。

- 画像を取得し、内容を解析する。
    <!-- 全角空白があるとpermissionの記載に一致せずに毎回実行許可が必要となるため -->
  - mdファイル名に全角空白「　」が含まれている場合は、`Bash`で「-」に置き換える(画像取得スクリプト)
    - 例: `記事　タイトル.md` → `記事-タイトル.md`
    - `bash mv "<元のファイルパス>" "<新しいファイルパス>"`
  - 画像ファイルは`Bash`で以下スクリプトを実行して画像を一時ファイルに保存する。(デバッグ用の`2>&1 | head -50`は本番実行時は利用しない)
  - `Read` ツールで `画像メタ情報保存パス` のファイルを読み込む
  - `local_path` があるエントリは、そのパスの画像ファイルを `Read` ツールで直接読み込んで内容を分析する
  - `local_path` が null（取得失敗）のエントリは `alt`・`context_before`・`context_after` で内容を推測する

```bash
 python <画像取得スクリプトのパス> "<markdownファイルのパス>" --output "<画像メタ情報保存パス>"
```

### 3. ファイル内容の整理と記事分割

手順２で取得した内容をもとに、以下の基準で記事を分割する。

#### 分割基準

- 内容を精査しZettelkasten方式で記事を分割する
- 分割した記事の内容が既存の記事(整理先フォルダの他ファイル)と重複していないか確認する（重複している場合は、既存の記事に内容を統合するか、分割した記事を削除する）

#### 記事内容

- Markdown含めコード例や記載例などは可能な限り記載する（例: AWSのIAMに関する記事なら、記事中のIAMのコード例を含める）
  (記事の内容を再現するために必要なコード例は必ず含める。コード例がない場合は、内容を理解するために必要なコード例を生成して含める)
- 既存記事を確認し、分割した記事内で既存記事と関連する内容・用語がある場合は内部リンク（例：`[[記事パス]]`）を追加し既存記事との関連付けを行う
- 記事内容の理解に役立つ画像の場合は、分割記事内にも必ず記載する。（例: `![画像説明](https://example.com/article)`）
- 分割記事内の画像は元記事を引用元として必ず記載する。（例: `出典: [引用元記事タイトル](https://example.com/article)`）

#### メタデータ等

- ファイル名は分割した内容を反映させる
- ファイルはテンプレート（`./05_Template/Article.md`）に基づいて作成する
- 各種メタデータ（タイトル、タグ、カテゴリ、サブカテゴリなど）を適切に反映させる（）
- 各記事には元ファイルのSourceタグのURLを以下に基づいて記載する
  - URLのドメインが以下の場合は、URLを記載せず「`引用元: サービス名`」（例： `引用元: NotebookLM`）と記載する
    - `notebooklm.google.com`
    - `gemini.google.com`
  - URLがない場合は記載しない
  - 記事本文の最後に「`引用元: [元記事タイトル](URL)`」の形式で記載する(例: `引用元: [引用元記事タイトル](https://example.com/article)`)
  - Sourceタグにも元ファイルのSourceタグのURLを記載する

#### ファイル保存先

- 分割記事に利用した画像は、`画像保存先` に格納する
- 分割した記事はパス設定に記載している`整理先`に格納する

#### 禁止事項

- フォルダは新規作成しない
- タグには空白を使用しない

### 4. カテゴリ・サブカテゴリ判定

#### カテゴリ

- 分割したそれぞれの記事の主要な技術領域を判定する。
- 判定したカテゴリのMOCファイルに記事のリンクを追加する。
- 適したMOCがない場合は新規作成する。

例:

- `AWS` - Amazon Web Services関連
- `GCP` - Google Cloud Platform関連
- `Azure` - Microsoft Azure関連
- `Terraform` - Terraform / IaC関連
- `Kubernetes` - Kubernetes / コンテナオーケストレーション関連
- その他、記事の内容に適した名前で新規作成

#### サブカテゴリ

カテゴリ内の具体的なサービスや分野を判定する。
判定したサブカテゴリはMOC内でカテゴリの下に階層化して整理する。

例:

- AWS → `IAM`, `EC2`, `Lambda`, `S3`, `S3Tables` 等
- GCP → `IAM`, `CloudRun`, `GKE`, `BigQuery` 等
- Terraform → `Basics`, `Modules`, `State` 等

### 5. Clip記事の削除

- 整理が完了したら、入力元フォルダ内の元のmdファイルを削除する。
- `画像情報保存パス` のフォルダを削除する。

### 6. 処理結果レポート

整理結果をテーブル形式で表示する。
